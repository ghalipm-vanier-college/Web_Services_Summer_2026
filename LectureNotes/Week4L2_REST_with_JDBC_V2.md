### Theory Notes

#### What is JDBC?
**JDBC (Java Database Connectivity)** is a Java API that enables Java applications to communicate with relational databases. It allows applications to:
* Connect to a database
* Execute SQL statements
* Retrieve, insert, update, and delete data

#### Why Do We Need JDBC?
Without JDBC, a Java application has no native pathway to understand a vendor-specific relational database. JDBC acts as a standardized bridge between Java applications and databases:

```text
Java Application ──► JDBC ──► SQL Server
```

### JDBC Architecture

```text
Java Application ──► JDBC API ──► JDBC Driver ──► SQL Server
```
### Main Components
 - **DriverManager**: Responsible for establishing database connections.

```java
 Connection conn = DriverManager.getConnection(url, user, password);
```
 - **Connection**: Represents an active connection session to the database.

```java
Connection conn;
```
 - **Statement**: Used to execute static SQL statements and return their results.
```java
Statement stmt = conn.createStatement();
```
 - **ResultSet**: A table of data representing a database result set, usually generated by executing a statement that queries the database.

```java
ResultSet rs = stmt.executeQuery(sql);
```

### SQL Server JDBC Driver

The SQL Server JDBC Driver allows Java applications to communicate with SQL Server.

### Maven Dependency
```xml
<dependency>
    <groupId>com.microsoft.sqlserver</groupId>
    <artifactId>mssql-jdbc</artifactId>
    <version>13.4.0.jre11</version>
    <scope>compile</scope>
</dependency>
```
### JDBC Workflow

A typical JDBC program sequentially follows these five steps:

```text
1. Open Connection ──► 2. Create Statement ──► 3. Execute SQL Query ──► 4. Process Results ──► 5. Close Resources/Connection
```

### Step 3: Enable TCP/IP via PowerShell

 - Open your Windows Start Menu and type PowerShell.

 - Right-click on Windows PowerShell and choose Run as administrator (Crucial!).

 - Copy, paste, and run this command block to enable TCP/IP:

```powershell
Import-Module SQLPS -DisableNameChecking -ErrorAction SilentlyContinue
$wmi = New-Object 'Microsoft.SqlServer.Management.Smo.Wmi.ManagedComputer' 'localhost'
$uri = "ManagedComputer[@Name='localhost']/ServerInstance[@Name='MSSQLSERVER']/ServerProtocol[@Name='Tcp']"
$tcp = $wmi.GetSmoObject($uri)
if ($tcp.IsEnabled -eq $false) {
    $tcp.IsEnabled = $true
    $tcp.Alter()
    Write-Host "SUCCESS: TCP/IP has been enabled for SQL Server!" 
} else {
    Write-Host "TCP/IP was already enabled." 
}
```
 - Run the following block to explicitly bind the instance to static port 1433 and restart the engine:

```powershell
$ipAll = $tcp.IPAddresses | Where-Object {$_.Name -eq "IPAll"}
$ipAll.Properties["TcpPort"].Value = "1433"
$tcp.Alter()
Restart-Service -Name "MSSQLSERVER" -Force
Write-Host "SUCCESS: Port set to 1433 and SQL Server restarted!" 
```

### Theory Notes
#### SQL Injection

 - SQL Injection occurs when untrusted user input is directly concatenated with raw SQL query strings.

#### ❌ Unsafe Example

```sql
String sql = "SELECT * FROM Student WHERE id=" + userInput;
```
If an attacker enters 1 OR 1=1 as input, the runtime interpreter evaluates:

```sql
SELECT * FROM Student WHERE id = 1 OR 1 = 1
```
Because 1=1 is always true, this query bypasses logic checks and exposes all user records.

### PreparedStatement

 - **PreparedStatement** neutralizes SQL Injection by pre-compiling the SQL query structure and handling user inputs strictly as parameters, never as executable code.

#### Safe Example

```java
// Define parameter placeholders using '?'
PreparedStatement ps = conn.prepareStatement("SELECT * FROM Student WHERE id=?");

// Safely bind values to parameters
ps.setInt(1, id);

// Execute query safely
ResultSet rs = ps.executeQuery();
```

#### CRUD Operations Reference


| Operation | SQL Command | JDBC Method |
| :--- | :--- | :--- |
| Create | `INSERT` | `executeUpdate()` |
| Read   | `SELECT` | `executeQuery()`  |
| Update | `UPDATE` | `executeUpdate()` |
| Delete | `DELETE` | `executeUpdate()` |

**DAO**: Data Access Object. 
 - It is a software design pattern that isolates the application's business logic from the persistence layer (the database).
 - What it Does?
   - Separates Concerns: Your core program logic does not need to know how data is stored.
   - Encapsulates Database Code: All raw SQL queries, connections, and result sets live inside the DAO.
   - Provides a Clean Interface: The rest of your application interacts with simple Java methods instead of database commands

#### Task 2 — Add Student (Create)

```java
public void addStudent(int id, String name) throws SQLException {
    String sql = "INSERT INTO Student VALUES(?, ?)";
    PreparedStatement ps = conn.prepareStatement(sql);
    
    ps.setInt(1, id);
    ps.setString(2, name);
    
    ps.executeUpdate();
}
```

#### Task 3 — Retrieve Student by ID (Read)

```java
public void getStudentById(int id) throws SQLException {
    String sql = "SELECT * FROM Student WHERE id=?";
    PreparedStatement ps = conn.prepareStatement(sql);
    
    ps.setInt(1, id);
    ResultSet rs = ps.executeQuery();
    
    while(rs.next()) {
        System.out.println(
            rs.getInt("id") + " " +
            rs.getString("name")
        );
    }
}
```

### Theory Notes

#### Architectural Evolution

#### Legacy Architecture (Static Response)
 
 - Previously, servlets processed calls by serving static, mocked strings: 
 ```text
 Postman ──► Servlet ──► Hardcoded JSON String
 ```
 -  `The Problem`: Data changes require code re-compilation. It does not reflect persistence-realities and is unusable for real production applications.
 
 
#### New Architecture (Data-Driven Web Service)
```text
Postman ──► Servlet ──► JDBC Bridge ──► SQL Server Database
```
 - `The Benefits`: Real-time persistence updates, dynamic state tracking, data scalability, and a fundamental transition into standardized REST API architectures.

#### Request Processing Flow
```text
Client Request ──► Servlet Handler ──► Database Query ──► Build JSON ──► Transmit Response
```
